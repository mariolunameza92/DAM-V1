Guía de decisión — cómo resolver cada tipo
⚠ A-type — clase HTML redundante
  • El CSS de contexto ya fija el tamaño con .selector .msi{font-size:Xpx}
  • La clase .sm / .xs en el HTML no tiene efecto visual
  • Fix A (recomendado): quitar la clase del JS — CSS es la fuente de verdad
  • Fix B: quitar el override CSS y usar solo clases HTML
  • Aplicar la misma opción a todos los A-type
🔴 B-type — bug real (inline vs clase)
  • El elemento tiene clase de tamaño (.sm/.xs) y style="font-size:Xpx" diferente
  • El inline siempre gana — la clase nunca tiene efecto
  • Fix inmediato: quitar el style="font-size:..." del JS
  • Si el tamaño inline es correcto, moverlo al CSS del contexto
  • Archivos: analytics.js → _buildSecuritySection(), _buildQualitySection()
🟣 S-type — fondo de icono inconsistente
  • Dentro del mismo card coexisten fondos con gradiente y fondos planos
  • El gradiente --an-grad-dark crea jerarquía visual no intencional
  • Regla propuesta: gradiente solo en KPI cards (punto focal primario)
  • En contextos secundarios (alert icons, mini cells dentro de card): solo fondos planos diferenciados por tono
  • Archivo CSS: analytics.css → reglas de .an-alert-icon, .an-mini, .an-badge

S4kc-blue, kc-green, kc-amber, kc-rose, kc-violet — todos producen el mismo CSS
Las 5 clases kc-* en analytics.css (líneas 176–180) tienen exactamente el mismo valor: background: var(--an-grad-dark); color: var(--on-accent). Los nombres implican diferenciación cromática (blue/green/amber/rose) que no existe en la implementación. No es un bug visual pero es deuda de mantenimiento: si en el futuro se quiere diferenciar por color, habría que editar 5 reglas que "debían" ser distintas.
