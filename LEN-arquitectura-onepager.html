02

El modelo: plataforma modular multi-tenant

Plataforma
Infraestructura compartida, no una app. Capacidades reutilizables entre mercados.
Modular
Módulos activables por tenant (enabled_modules). Nada hardcodeado a una vertical.
Multi-tenant
Muchos clientes, datos 100% aislados. Ningún query cruza tenants.
N aplicaciones
Cada vertical = una subversión (schools / dam_b2b) con su UX y marca.

05

Flujos críticos

UC-01Consentimiento por hijo
Padre firma → PDF inmutable → si rechaza módulo crítico ⇒ alumno a blacklist. Sin consentimiento vigente = acceso bloqueado.
UC-02Detección al subir (costo)
Match solo contra blacklist (~30), no contra el padrón (~1500) → ~50× más barato. Lógica invertida: no en blacklist = autorizado.
UC-03Revocar consentimiento
Nuevo PDF versionado → blacklist en milisegundos → fotos publicadas pasan a hidden retroactivo.
UC-05Búsqueda DAM multi-filtro
Persona × fecha × evento × logo, query ordenado por selectividad → export masivo · auditado.
06

No negociables

base de credibilidad enterprise
① Aislamiento multi-tenant
Prioridad uno, por encima de performance y costo. "Prefiero que se caiga mil veces antes de un cruce entre tenants." Ningún query, endpoint ni estructura lo viola.
② Privacy by Design
Compliance = feature comercial. Ley 29733 (PE) · 21.719 (CL) · GDPR como techo. Embeddings = dato biométrico cifrado. Audit inmutable ≥24 meses. ARCO propagado a backups + downstream.
③ Async + API-ready
Procesamiento por cola, idempotente y elástico para picos. Toda capacidad core exponible vía API con token, cuotas y billing — se diseña ahora, se activa después.
07

La regla que guía cada decisión

decisión arquitectónica nº 1
¿Esto vive solo en una aplicación (ej. Schools), o vive en la plataforma común que sirve a todos los verticales?
Si la lógica podría aplicar a otros verticales pero la escribís "solo para colegios" → refactorizá a la plataforma común. Solo lo genuinamente específico de la vertical (cuentas padre→hijo, calendario escolar) vive en la capa de aplicación.
